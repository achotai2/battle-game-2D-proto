Refactor my worker/job system in Godot 4 so task acquisition/execution is moved OUT of TacticalWorker and into a dedicated worker-task node. The new node should: advertise availability, communicate with CastleJobBoard, accept/abandon jobs, command AgentMovement to move to the site, and once in range LOCK the worker into working (no meander flicker). Fix the current bug where workers arrive at the site but start meandering, run back, meander again, and only occasionally work.

CONTEXT / CURRENT PROBLEM
- Workers currently request jobs / manage sites in TacticalWorker and also rely on Movement’s idle-meander.
- Result: once a worker reaches a site, it may briefly have “no movement order” (or loses it), so Movement auto-meanders (prio 0), causing the worker to drift away and re-approach repeatedly.
- I want: once within work range, worker should STOP moving and WORK repeatedly until the site is complete or invalid. No meander once job is accepted.
- I want to generalize the tasker so that it can be used on other minions, not just workers. 

NEW DESIGN
Create a new node script to attach to worker units (agents):
- class_name MinionTasker (or MinionJobAgent)
- Responsibilities:
  1) Register with a JobBoard (from castle) as an available worker. Removing communication with job task board away from tactical_worker.gd
  2) Request a job when idle (with a think timer).
  3) When assigned a site, command movement to go to that site.
  4) When in range, stop issuing movement commands and trigger a work action loop.
  5) Hold a “job lock” freeze movement order so the unit never falls into idle-meander while it has a job.
  6) Release job when complete/invalid and return to idle.

CRITICAL BEHAVIOR CHANGE (to fix meander flicker)
- While the worker has an active job (even after arriving), MinionTasker must keep a movement order active at a higher priority than meander (priority=5).
- When in work range, it should clear movement velocity (command_move_velocity(Vector2.ZERO, 5)) OR clear order with higher prio so it does NOT drop to idle-meander. Easiest: keep an explicit “HOLD” order at priority 5 while working, or keep sending zero-velocity commands at priority 5.
- Also: when in work range, worker should freeze via Movement’s lock system and run the work animation/hit loop:
  - movement.start_work() (uses LOCK_WORK, freezes movement)
  - repeat on a timer until site completes
- While frozen for work, do NOT allow Movement to switch to meander. Freeze already makes velocity zero; but ensure we still keep “job active” state so on unfreeze it resumes work instead of meandering.

IMPLEMENT MinionTasker.gd
- Node attached to worker agent. It should export:
  - @export var agent: Node2D
  - @export var movement: AgentMovement
  - @export var castle: Node2D
  - @export var work_range: float = 28.0
  - @export var work_interval: float = 0.35
  - @export var work_amount: float = 1.0
  - @export_range(...) var think_interval: float = 0.25
- Internal:
  - var _job_board: CastleJobBoard (or Node)
  - var _site: Node2D = null
  - var _state enum { IDLE, MOVING, WORKING }
  - timers: think_timer, work_timer
  - const JOB_PRIORITY := 5

- Responsibilities:
  1) set_castle(castle): resolve job board (castle.get_job_board or castle.get_node("JobBoard") or castle itself if it has request_job)
  2) IDLE:
	 - periodically request job
	 - if got a site: set _state=MOVING, command_move_to_position(work_pos, JOB_PRIORITY)
  3) MOVING:
	 - keep commanding move_to_position(work_pos, JOB_PRIORITY) on think ticks (refresh) so order never drops.
	 - if within range -> transition to WORKING:
		- command_move_velocity(Vector2.ZERO, JOB_PRIORITY) (so we keep an active order)
		- start work loop: work_timer.start()
  4) WORKING:
	 - stop sending path commands; keep “hold position” command each think tick: command_move_velocity(Vector2.ZERO, JOB_PRIORITY)
	 - On work_timer:
		- if site invalid or done -> release and go IDLE
		- else call movement.start_work() (freezes movement + plays animation). If start_work() returns true:
			- when animation finishes Movement unfreezes automatically via its internal signals
			- then apply_work() once per timer tick (or apply_work only if start_work succeeded)
		- Ensure worker remains in WORKING state until complete; never allow meander while WORKING.
  5) Completion/abandon:
	 - job_board.release_job(_site, agent)
	 - _site=null; state=IDLE; stop work_timer
	 - optionally clear movement order by issuing command_move_velocity(Vector2.ZERO, JOB_PRIORITY) briefly or clear with priority > current (e.g. 6) so Movement can meander again.

- Site helpers:
  - func _get_work_pos(site)->Vector2
  - func _site_needs_work(site)->bool
  - func _apply_work(site, amount)->void

IMPORTANT: Prevent auto-meander while working
- While _site != null, MinionTasker MUST maintain an active order at priority 5 (either move_to_position while traveling or zero-velocity hold while working).
- Do NOT call movement.clear_movement_order(5) because it only clears if > current; use priority 6 for clears if needed.
